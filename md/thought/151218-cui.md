#### テキストデータ

- かつて人々は、音声によってコミュニケーションを図り、自然言語を記号列として発達させていった。また、近年、言葉を書き記すことを始めた。歴史的には音声に由来する自然言語は、ほとんど 1 対 1 に、直線状に表現できた。
- 記録は、社会的にも便利だったが、暗算だけで数を計算するのが難しいように、個人についても計算力を拡大した。
- 人々が発明したコンピュータは、2 進数の列を表すことを得意としていた。それを最も単純に自然言語に結びつける方法として、8 ビットを 1 文字に結びつけるような方法が開発された。ASCII という文字エンコーディングが標準となり、さらには UTF-8 という文字エンコーディングが広く用いられて、世界中に実用されるほとんどの文字を表せるようになった。
- また、数の処理は、例えば 8 ビットによって 0 から 255 を表し、加算や乗算をすることが行われた。
- しかしまた、例えば ASCII コードによって、8 ビットを 3 つ使って文字通り "255" と表すこともできる。これは、低レベルにおける計算のパフォーマンスのためにはマイナスだが、自然言語と同じエンコーディングで単純に、あるいは混ぜて表現することもでき、人間にとっての視認性がいい。
- そのため、テキストデータは、汎用性において高い地位を得た。
- Unix においては、標準入力と標準出力というものを考え、特にそこでテキストデータを用いることで、プログラムの動的な構成を支援した。
- また、タグによって任意の構造を表すためには、XML や HTML が発明されて広く持ちいられた。

#### GUI は糞か

- 歴史的なコンピュータは、文字だけで画面を表示し、キーボードだけで操作した。やがてコンピュータは進歩して、任意な画像で表現し、マウスやタッチによって、直感的で素早い入力を許すようになった。つまり、CUI から GUI へと進歩した。
- しかし、GUI に表示されたファイルを操作する場合、マウスでクリックしたりダブルクリックしたり、操作の種類が制約される。一見、使いやすいが、それ以上に効率化することが難しい。
- また、指を指して位置を指定したり、音声で文字列を入力する場合、正確性が損なわれやすい。
- コンピュータは、低レベルな部分を完全に隠蔽してバグなく動作するほど進歩していない。ゆえに、データについても、可能なら、人間に可読なテキスト形式を求める圧力が強い。入力も、正確である必要がある。
- また、GUI では、本質的な要素だけではなく、演出的な要素が加わりやすい。model-view-controller なる形で言えば、view が厚くなる。それは、記録や自動化をしにくくする。
- 複雑な操作は複雑に行うしかないが、CUI で行える操作まで GUI で行うと、操作は限られ、例えば標準入出力によるプログラム同士の接続がしにくくなる。すると、個々のプログラムに機能が集中する。
- プロプライエタリなプログラムが囲い込みを行うためには、それは都合が良い面があるだろう。しかし、コンピュータらしい考え方からは離れてしまうことになるかもしれない。プログラムの合成をユーザに許してこそ、ユーザは知性の拡張にコンピュータを生かせる。その自由性が用いられないとしても、その自由性が存在するほうが健全である。
- 例えば、ls コマンドで得たファイルのリストから、grep コマンドで特定の条件を満たすものを抜き出し、リダイレクトでテキストファイルに出力をする。あるいはまた、それを自動化する。CUI では初歩的でしかないたったそれだけの操作が、GUI では難しい。
- その意味で、GUI はかえって不便である。だから、CUI は陳腐化しない。
- 道具は、不可能を可能にする。ゆえに道具は、考え方を規定する。不自由な環境に置けば、才能のある子も猿に留まる。ゆえに、CUI とそれについての助言は、一般にアクセス可能であるべきだと思われる。
- 新しい技術を使って刺激的な映像をもたらす映画は誰しもに面白く、その意味で、CUI はとてもつまらない。そうでなくても、GUI の有用性は様々な実務においてとうに明らかである。インタフェースの情報量が少なく、直感的でなくて、例えば自動車のスクリーンが CUI で、現在の速度を調べるコマンドや、左右に曲がるコマンドがあったなら、とても怖い。
- しかし、パソコンとして一般的である Windows OS のコマンドライン環境の質が Linux に比べてあまりに低く、多くのユーザにとって実際、Windows と Linux の間の垣根が高いのを感じると、教育上かなり好ましくない気がする。ファイル操作などの基本的な処理は、かなり昔に枯れており、重要であるわりに陳腐化せず、したとしても遅い。
- 例えば、ls コマンドや cd コマンドは、きっとかなり昔から ls や cd という名前で、機能もほとんど変わらないだろう。一見、覚えることが膨大にあるようだが、とても合理的に整理されている上に、陳腐化しないから、少しずつ慣れていく価値はある気がする。

#### shebang

- Linux には、shebang という機能がある。shebang とは何者だろう？ 例えば、Python 言語で書かれたプログラムのテキストファイルの 1 行目にニョロニョロと書くと、コマンドラインからそのファイルを名指しするだけで実行できる。
- とてもいい！　インタープリタの用意されている多くの言語で利用できて、私が勉強したいと憧れている Haskell 言語でも使えるようだ。
- スクリプトというと、bash スクリプトというイメージがあったが、Haskell スクリプトとでもいうものが使えるとなると、何やらかっこ良い感じがする。
- 世界中のプログラムを Haskell スクリプトで置き換えたい気がする。しかしそれは、無知だから言えることで、色々な意味で無理だろう。
- そんな感じで、Linux に触れて、インタプリタ言語への印象が良くなった。コンパイル面倒くさい。というか、ソースコードと実行ファイルが分離されているなんて気持ち悪い。
- 標準入出力で、プログラム同士をコンパイルなしで動的にプチプチ繋げるのも驚くが、スクリプト（特に Haskell スクリプト）という概念も与えられると、世界観を更新するくらいの感じがする。
- 実際何がしたいわけでもないが、そういうことができるのかなと思って、少し満足感がある。
- clisp というのもいいかもしれない。pascal もできた。

#### emacs

- 先日、vim を勉強していて、emacs に転向した。
- vim は、モードを切り替えるのが面倒くさい気がしたのと、デフォルトの設定だと論理行で上下移動することに違和感があった。キーバインドのカスタマイズは極力、避けたい。
- emacs も、ワードラップができない！？ぽくてびっくりしました。
- CapsLock を Ctrl にしていたけど、やめました。tty1 で CapsLock を押してしまう自分がダサかったから、かな。癖になりすぎる。Linux に触るからには、tty1 だけで生きていけるようになりたい。
- 例えば emacs を、CUI で使うか、GUI で使うか、悩んでいる。
- ブラウザがタブだらけになるのは、今の感じだと、避けられない。ターミナル（エミュレータ）がタブだらけになるのも、避けられないだろう。その上で、emacs の GUI もタブだらけになると、頭が混乱するし、遷移が面倒くさい。なおモバイルノートパソコンの画面を想定している。
- なので、emacs を CUI で（ターミナルの中で）使って、タブだらけのブラウザと、タブだらけのターミナルだけ、というのを基本にやっていきたいな、と少し思いました。
- そもそもなぜ vim や emacs を使うかって、tty1 で普通に起動するのがかっこいいと思う。依存しなくてもよいものに依存していないことに力を感じる。
- まあ、コピー＆ペーストは、Ctrl+C、Ctrl+V であってほしいし、それしか覚えられないので、辛いといえば辛い。デフォルトでそんな操作な CUI テキストエディタが vim や emacs よりもシェアを持っていたなら、迷わずそれを使う気がするが、そんなものはないだろう。
- しかし、今の emacs の設定だと CUI だと背景が不透明でダサく、GUI だと半透明でかっこいい。なので GUI で使っていたが、ウィンドウ 3 つクルクルしてしばしば間違えるのが面倒になった。（Alt+Tab でクルクルしているのである。）
- なので、GUI に対する CUI の長所や、より「普通」な操作のテキストエディタではなく emacs や vim を使う長所について、現在時点で少し整理してみようと思いこの文書を書き始めた。
- まとめ。ターミナルはどうせ開いておかなきゃいけないし、ウィンドウが増えると操作が煩わしいから、CUI なテキストエディタを使うべき。また、テキストエディタの選択肢は vim か emacs であり、emacs のほうがよい。
- emacs が間違ったインデント支援をしてくるのがとても嫌なのだが、そのうちどうにかする予定である。新しい行に提案してくれるインデントが間違っていて、直して、その行を書き終えて、改行を押すとせっかく直したインデントをばきっと壊してぐちゃぐちゃにしてくれる感じ。
- スクリプトファイルをいくらか触ってみながら、自分にとってのスクリプトファイルの実用性について調べたい。CUI 環境の長所を考えるためには、その辺りが大切そうである。
